# Cost Simulation A — Bottom-Up Operator Cost Model

**Author:** Independent infrastructure-cost analyst
**Date:** 2026-06-16
**Scope:** Cost borne by the *platform operator* of the Hermes agent-orchestration SaaS.
**Method:** Bottom-up. Counts derived directly from the code (`convex/http.ts`, `connector/control_plane/agent_runtime.py`, `convex/a2a.ts`, `convex/engine.ts`, `convex/lib/guards.ts`, `convex/lib/metering.ts`, `convex/lib/events.ts`, `convex/crons.ts`). No reliance on any other model's numbers.

---

## 0. What is and isn't in scope

**IN (operator pays):**
- **Convex** — DB storage + database I/O bandwidth, function calls (queries/mutations/actions), action compute (GB-hr), HTTP actions, crons, scheduler, vector search.
- **Vercel** — thin Next.js UI (Pro seat + usage).
- **Clerk** — auth (MAU-priced).
- **OpenAI embeddings** (optional) — only the embedding call for RAG indexing/query. Small.

**EXCLUDED (end users pay their own — called out explicitly):**
- **Agent compute** — `agent_runtime.py` runs on the *user's* machine/VM/Cloudflare container. The 2-second loop, the LLM client, the Python process: all on user infra. The operator never pays for it.
- **User LLM tokens** — `llm_respond()` calls `api.anthropic.com` / `api.openai.com` with the **user's** `ANTHROPIC_API_KEY` / `OPENAI_API_KEY` (see `agent_runtime.py` lines 43–74). Those tokens are billed to the end user, not the operator.

> The operator's cost is therefore almost entirely **Convex serverless traffic generated by the polling connector**, plus fixed Vercel/Clerk seats. This is the central finding.

## 2. Confirmed system mechanics (read from code)

| Mechanic | Code evidence | Convex billable units per occurrence |
|---|---|---|
| **Idle poll loop** | `agent_runtime.py` L110–133: `while True:` → `workflow_inbox()` + `a2a_inbox()` then `time.sleep(2.0)` | 2 HTTP actions / 2s. Each HTTP action (`/workflow/inbox`, `/a2a/inbox`) runs `authAgent` (1 `runQuery`) + 1 `runMutation` (`claimSteps` / `pullInbox`). Counted below as **2 function calls per HTTP action** (the action + its dominant child). |
| **Heartbeat** | `agent_runtime.py` L112: every 30s → `/connector/heartbeat` | 1 HTTP action → `authAgent` query + `recordHeartbeat` mutation = 2 fn calls. |
| **A2A message route** | `convex/a2a.ts` `route()` L144–250 | **6 DB writes**: `threads` insert/patch (1), `a2aMessages` (1), `messages` (1), `activity` (1), `workEvents` via `recordWorkEvent` (1), `usage` via `recordUsage` (1). Plus guard reads. |
| **A2A guards** | `convex/lib/guards.ts` | `assertWithinDailyBudget`, `assertNotLooping`, `assertRateLimit` each `.collect()` `a2aMessages` over a time window. `assertWithinBudget` + `recordUsage` `.collect()` `usage` over the month. **Read bandwidth grows with message volume in the Space** (O(messages) per send). |
| **Workflow step** | `convex/engine.ts` `advanceRun` + `completeStep` + `claimSteps` | dispatch patches step + run + activity; completeStep patches step + run + `workEvents` + `usage` + `scheduler.runAfter`. **≈6–10 writes + 2–3 scheduler invocations** per step. |
| **MCP / tool call** | `convex/http.ts` `/integrations/execute` → `runAction` (fetch) + recordCall | 1 action (GB-seconds of compute) + ~3 writes. |
| **Crons** | `convex/crons.ts` | `trigger tick` + `health sweep` every 1 min (paginated), `daily digests` daily. **Operator-wide fixed cost, not per agent.** |
| **RAG** | `convex/http.ts` `/context/search` → `memories.retrieveForConnector`; `agent_runtime._augment` L93–102 | 1 `vectorSearch` (action, query-GB) + 1 embeddings call per query. Called on **every** workflow step and **every** A2A reply (`_augment` wraps both). |

**Function-call counting convention:** Convex bills each query, mutation, *and* action as a separate function call. An HTTP action that calls one mutation = 2 function calls. I use **2 fn calls per idle HTTP action** (action wrapper + the query/mutation it dominates). The `authAgent` query technically makes idle polls 3 calls; I keep 2 as the *expected* case and surface 3 in the **high** range.

## 3. Per-always-on-agent monthly volumes

30-day month = **2,592,000 seconds**.

### 3a. Idle traffic (the floor — present even if the agent does zero work)

| Source | Rate | HTTP actions/mo | Function calls/mo (×2) |
|---|---|---:|---:|
| Poll `/workflow/inbox` + `/a2a/inbox` | 2 actions / 2s | 2,592,000 | 5,184,000 |
| Heartbeat `/connector/heartbeat` | 1 / 30s | 86,400 | 172,800 |
| **Idle total** | | **2,678,400** | **5,356,800** |

So **one idle always-on agent burns ~5.36M Convex function calls/month** — purely from existing and polling.

### 3b. Active-work traffic (per unit of real work, on top of the floor)

Assume an *expected* moderately-busy agent: **200 A2A messages/mo, 100 workflow steps/mo, 50 MCP calls/mo, 300 RAG queries/mo** (300 = 200 A2A replies + 100 steps, since `_augment` fires on both).

| Work unit | Count/mo | Fn calls each | DB writes each | Fn calls/mo | Writes/mo |
|---|---:|---:|---:|---:|---:|
| A2A message (route + 3 guard collects + 4 internal queries) | 200 | ~8 | 6 | 1,600 | 1,200 |
| Workflow step (advance+complete+claim+scheduler) | 100 | ~12 | 8 | 1,200 | 800 |
| MCP/tool call (action + recordCall) | 50 | ~5 | 3 | 250 | 150 |
| RAG query (vectorSearch action + embeddings) | 300 | ~3 | 0 | 900 | 0 |
| **Active subtotal** | | | | **~3,950** | **~2,150** |

**Active work is a rounding error next to idle polling** (~3,950 vs 5,356,800 function calls — **0.07%**). The guard `.collect()` reads do scale with Space message volume, but at these volumes the read bandwidth is well under 1 GB/agent/mo. Embeddings for 300 RAG queries at ~500 tokens each ≈ 150k tokens ≈ **$0.003/agent/mo** — negligible.

### 3c. Per-agent cost table (expected case, 2s poll)

| Cost component | Volume/agent/mo | Unit price | Cost/agent/mo |
|---|---:|---:|---:|
| Function calls (idle 5.357M + active ~0.004M) | 5.361M | $2 / 1M (overage) | **$10.72** |
| Action compute (HTTP actions ~50ms @128MB) | ~4.7 GB-hr | $0.30 / GB-hr | $1.40 |
| Database I/O (~1 KB/idle call + active reads) | ~3 GB | $0.20 / GB | $0.60 |
| Vector search query-GBs | <50 query-GB | incl. / $0.50/1k | ~$0.00 |
| OpenAI embeddings | ~0.15M tok | $0.02/1M | $0.003 |
| Storage (DB rows accrue) | <0.5 GB | $0.20/GB-mo | ~$0.10 |
| **Marginal Convex / agent (at overage)** | | | **≈ $12.83** |

> These are **marginal** (overage) prices. The first few agents are absorbed by the included Pro quota (see §4) and cost effectively $0 marginal until the quota is exhausted. Function calls dominate every other line by ~8×.

## 4. Headline outputs

**(a) Cost per always-on agent / month**
- **Within quota:** ~$0 marginal (fixed $25 Convex + $20 Vercel + $0 Clerk covers the first batch).
- **Beyond quota (marginal):** **≈ $12.8 / always-on agent / month**, of which **$10.7 is function calls alone**.

**(b) Dominant cost driver**
**The 2-second idle poll loop's function calls.** 5.18M of the 5.36M function calls per agent (97%) come from polling `/workflow/inbox` + `/a2a/inbox` every 2s. This single behavior is ~84% of marginal per-agent cost. Everything else — action compute, I/O, RAG, embeddings, real A2A/workflow work — is in the noise.

**(c) Always-on agents that fit in the included Pro quota**
Binding constraint is **function calls (25M included)**:
- 25,000,000 ÷ 5,356,800 ≈ **4.7 idle agents** before function-call overage.
- Action compute (250 GB-hr) wouldn't bind until ~54 agents; DB I/O not until well beyond. **Function calls are the cliff, and it arrives at ~4–5 agents.**
- *(If the 250M-included figure from the aggregator is correct instead, this becomes ~46 agents. Verify.)*

**(d) Marginal $/agent beyond the quota**
**≈ $12.8 / agent / mo** (≈ $10.7 function calls + ~$1.4 action compute + ~$0.7 I/O/storage/RAG). Round figure: **~$13/always-on agent/month** once past the included quota.

## 5. The key lever — polling interval

Idle function calls = `[2·(seconds/interval) + (seconds/30)] × 2`.

| Polling model | Idle fn calls/agent/mo | Marginal fn-call cost/agent/mo | Idle agents per 25M quota | % cost reduction vs 2s |
|---|---:|---:|---:|---:|
| **2s poll (current)** | 5,356,800 | $10.72 | 4.7 | — (baseline) |
| **10s poll** | 1,209,600 | $2.42 | 20.7 | **−77.4%** |
| **Event-push / SSE** (heartbeat only) | 172,800 | $0.35 | ~145 | **−96.8%** |

> **Killing the 2s poll loop is the single highest-leverage cost action available.**
> - Going **2s → 10s** cuts idle function-call cost **~77%** and ~4×'s the agents-per-quota (≈5 → ≈21).
> - Going to **event-push** (the code already has an SSE path: `/a2a/stream` in `convex/http.ts` L565, bounded ~20s reconnect) cuts idle cost **~97%** and fits **~145 idle agents** in the same quota. Heartbeat then dominates the residual — widening it 30s → 60s+ shaves further.
> - Note: SSE reconnects are themselves HTTP actions, and each `/a2a/stream` connection loops `pullInbox` once/sec for ~20s internally (L576–589). If naively left as a 20s-window reconnect-immediately loop, it can be **as expensive as polling**. The win requires true long-lived/long-poll connections with a real idle timeout, not tight reconnect. Model the SSE residual conservatively.

## 6. Total operator bill — worked scenarios (expected case)

Fixed: Convex $25 (1 seat) + Vercel $20 (1 seat) + Clerk $0 (free <10k MAU) = **$45/mo floor.**

| Always-on agents | Poll = 2s | Poll = 10s | Event-push |
|---:|---:|---:|---:|
| 5 | ~$45 (just at quota) | ~$45 | ~$45 |
| 25 | $45 + ~$258 overage = **~$303** | $45 + ~$10 = **~$55** | ~$45 |
| 100 | $45 + ~$1,219 = **~$1,264** | $45 + ~$185 = **~$230** | $45 + ~$0 = **~$45–50** |

Overage = (agents − quota-capacity) × marginal. At 2s polling the business becomes function-call-bound almost immediately; at event-push it stays on the flat $45 floor up to ~145 agents.

## 8. Recommendations (analyst view)

1. **Verify Convex's included function-call number on the live billing page before any pricing decision.** The whole economic model hinges on 25M vs 250M.
2. **The 2s poll loop is the business's cost problem, full stop.** It is not the agents' real work that costs money — it's them asking "anything for me?" 43,200 times a day each. Move to the existing SSE/event-push path (`/a2a/stream`) with genuine long-lived connections, or at minimum back off the poll to 10s+. Expect **~77–97% reduction** in marginal per-agent cost.
3. **Price the product per always-on-agent-hour**, since cost is dominated by idle uptime, not activity. At 2s polling the true marginal cost is ~$13/agent/mo; charge with margin above that or fix the poll loop first.
4. **Agent compute and LLM tokens are correctly externalized** to end users — the operator's exposure is bounded and predictable (mostly Convex serverless + flat seats). Keep it that way.
