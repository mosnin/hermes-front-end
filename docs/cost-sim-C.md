# Cost Simulation C — Sensitivity & Architecture Levers

**Author:** Independent cost analyst (analysis C)
**Date:** 2026-06-16
**Scope:** OPERATOR-side infrastructure cost of running the Hermes agent-orchestration / comms layer.
**Stance:** Decision-grade and skeptical. Figures derived independently from code mechanics + current vendor pricing. I did not reuse any other analyst's numbers.

---

## 0. Executive framing (what this document answers)

> *"What are the cost levers, and where are the cliffs?"*

The operator pays **Convex + Vercel + Clerk** (+ optional embeddings). Users pay their own **agent compute + LLM tokens** — those are explicitly **excluded** here (the runtime calls `api.anthropic.com` / `api.openai.com` directly from the user's machine; see `connector/control_plane/agent_runtime.py:46-74`, and the LLM key lives in the user's env, never the operator's).

The workload is **ongoing multi-agent outreach**: agents run continuously, coordinate via A2A, and drive MCP tool chains (contact-lookup → AgentMail → MiniChat → Calendly). The defining property is that **agents are always-on and poll even when idle**.

**Headline finding:** The dominant operator cost is **Convex function calls generated by the idle poll loop**, not message volume, not bandwidth, not Clerk. An always-on agent doing *nothing* still burns Convex function calls 24/7. This is the cost engine and the first cliff.

## 1. Pricing inputs (cited, dated)

All retrieved 2026-06-16. Where a vendor's primary doc was not directly fetchable, I used the makerkit pricing calculators (which mirror the vendor pricing pages) and flagged the one gap.

### Convex — Professional plan
- **Base:** $25 / developer / month (per seat).
- **Included / month:** 50 GB DB storage, 50 GB DB bandwidth (I/O), 50 GB egress, **250M function calls**.
- **Overage:** function calls **$2 / million**; DB I/O **$0.20/GB**; storage **$0.20/GB**; egress **$0.12/GB**; action compute **~$0.30–0.33 / GB-hr**.
- Sources: [makerkit Convex calculator](https://makerkit.dev/pricing-calculator/convex), [toolradar Convex pricing](https://toolradar.com/tools/convex/pricing), [Convex docs — functions](https://docs.convex.dev/functions/actions).
- **Gap I am flagging:** Convex's public calculator explicitly says function-call counting is *"too app-specific to model"* and does **not** state whether an HTTP action that internally calls a mutation counts as **1** call or **2** (action + mutation). This matters enormously here (see §2). I model it as a **range** and treat "2 per poll round-trip subcall" as the expected case, because the code path is `httpAction → ctx.runMutation(...)` (e.g. `convex/http.ts:218`, `:235`) and Convex bills query/mutation/action executions as function calls.

### Vercel — Pro plan
- **Base:** $20 / user / month, includes a **$20 usage credit**.
- **Included:** 1 TB bandwidth, 1M function invocations.
- **Overage:** bandwidth **$0.15/GB** (US), function invocations **$0.60 / million**.
- Sources: [makerkit Vercel cost](https://makerkit.dev/blog/saas/vercel-cost), [costbench Vercel](https://costbench.com/software/developer-tools/vercel/).
- **Relevance:** Low. Vercel serves the *dashboard* (humans), not the agent poll traffic — agent traffic goes to Convex HTTP actions at `*.convex.site`, not Vercel. Vercel scales with operator-staff/customer dashboard usage, which is tiny vs. agent poll volume.

### Clerk — Pro plan
- **Base:** $25 / month, includes 10,000 MAU; **$0.02 / MAU** above 10k.
- Source: [Clerk blog — new pricing](https://clerk.com/blog/new-pricing-plans), [checkthat Clerk pricing](https://checkthat.ai/brands/clerk/pricing). (One Feb-2026 source claimed 50k free MAU; I use the conservative 10k figure.)
- **Relevance:** Low and bounded. MAU = human operators/customers logging into the dashboard, **not** agents (agents authenticate with bearer connector tokens via `authAgent`, `convex/http.ts:16-25` — not Clerk identities). Even 10,000 human seats = only ~$25/mo. Clerk is effectively a fixed cost until the operator has tens of thousands of *human* users.

### Embeddings (optional)
- `/context/search` (`convex/http.ts:437`) runs `memories.retrieveForConnector` (vector RAG). If the operator pays for embedding generation, it's a per-memory-write + per-query cost. The runtime calls `context_search` on **every** workflow step and **every** A2A reply (`agent_runtime.py:120,127` via `_augment`). Modeled as an optional lever in §6, not the base case.

## 2. Cost mechanics from the code (what actually generates load)

The always-on runtime loop (`agent_runtime.py:104-133`) per iteration does:

| Action | HTTP endpoint | Convex work triggered | Runs even when idle? |
|---|---|---|---|
| `workflow_inbox()` | `POST /workflow/inbox` | `httpAction` → `engine.claimSteps` (**mutation**) | **Yes** |
| `a2a_inbox()` | `POST /a2a/inbox` | `httpAction` → `a2a.pullInbox` (**mutation**) | **Yes** |
| `heartbeat()` (every ~30s) | `POST /connector/heartbeat` | `httpAction` → `agents.recordHeartbeat` (**mutation**) | **Yes** |

Loop cadence: `time.sleep(2.0)` → ~**0.5 polls/sec** → **2 inbox round-trips every 2s**.

**Function-call accounting per idle poll cycle (2s):**
- 2 HTTP actions (`/workflow/inbox`, `/a2a/inbox`)
- 2 internal mutations (`claimSteps`, `pullInbox`)
- = **2 actions + 2 mutations**. Under the "action+subcall both billed" model: **~4 function calls / 2s**. Under the optimistic "HTTP action subcall is free" model: **2 / 2s**.

Plus heartbeat: 1 action + 1 mutation every 30s ≈ **2 function calls / 30s** ≈ negligible vs. the inbox loop (the heartbeat is **~7%** of idle traffic — note this, it changes which lever matters).

**Per idle agent, per day** (expected 4-calls/cycle model):
- Poll cycles/day = 86,400 / 2 = **43,200**
- Inbox function calls = 43,200 × 4 = **172,800**
- Heartbeat function calls = (86,400/30) × 2 = **5,760**
- **≈ 178,560 function calls / idle agent / day** ≈ **5.36M / agent / month**

That is the load floor *before any actual work*. Confirmed hypothesis: **the idle poll loop dominates.**

**Why messages are cheap by comparison:** an A2A `send` (`a2a.ts:285-308 → route`) writes ~5 rows (a2aMessages, messages, activity, work event, usage) + runs 4 guards. Call it ~10 function-call-equivalents per *real* message. But real messages happen on the order of tens-to-hundreds per agent per day; idle polls happen ~43,200 times per day. **Idle polling out-volumes real work by 100×–1000×.**

**The guard `.collect()` amplifier (read-bandwidth, not call-count):** `runGuards` runs on *every* send and calls three functions that each do `.collect()` over a growing time window on `a2aMessages`:
- `assertWithinDailyBudget` — `.collect()` over **24h** of messages (`guards.ts:36-41`)
- `assertNotLooping` — `.collect()` over **60s** (`guards.ts:62-67`)
- `assertRateLimit` — `.collect()` over **60s** (`guards.ts:106-111`)
- `assertWithinBudget` — `.collect()` over the **whole month** of `usage` rows (`guards.ts:126-131`), and `recordUsage` re-runs the *same* monthly `.collect()` on every write (`metering.ts:50-55`).

These are indexed scans (good — `by_space_time`), but `.collect()` reads **every matching row** into memory. The 24h-window and month-window scans grow **linearly with message volume per space**. At high A2A volume this is a **DB-bandwidth (I/O) and action-compute** blowup that is quadratic in disguise: cost per message × messages, where cost-per-message itself rises with volume. This is the *second* cliff and the one that bites multi-agent "chatty coordination" spaces hardest.

## 3. Cost drivers, ranked

| Rank | Driver | Scales with | Cliff vendor | Why it dominates |
|---|---|---|---|---|
| **1** | **Idle poll loop** (`/workflow/inbox` + `/a2a/inbox` every 2s) | # always-on agents × (1/poll-interval) | Convex function calls | 43,200 cycles/agent/day regardless of work done |
| **2** | **Guard `.collect()` window scans** | (messages/space)² in effect | Convex DB I/O + action-compute | Re-scans growing windows on every send; month-window scanned twice per write |
| **3** | **Real A2A / workflow traffic** | messages/agent/day | Convex function calls + I/O | ~10 fn-calls + 5 writes per real message |
| **4** | **Embeddings / RAG** (optional) | every step + every A2A reply | embedding vendor + Convex egress | `_augment` called on all work |
| **5** | **Vercel** (dashboard) | human dashboard usage | Vercel bandwidth/invocations | decoupled from agent volume |
| **6** | **Clerk** (human auth) | human seats (MAU) | Clerk MAU | agents don't use Clerk; flat until 10k humans |

The top two are **Convex**. For this workload, **operator cost ≈ Convex cost** to first order.

## 4. Sensitivity tables

**Convention:** function calls (FC) modeled at expected case = **4 FC / poll cycle** (2 actions + 2 mutations) + heartbeat. Convex overage = **$2 / million FC** beyond the 250M/month Pro pool. Base seat cost ($25 Convex + $20 Vercel + $25 Clerk = **$70/mo fixed**) is held constant and excluded from the marginal tables below unless stated.

### 4.1 Poll interval (holding agents = 100, idle)

Monthly inbox FC/agent = (2,592,000 cycles/mo at 1s) scaled by interval, × 4 FC. (Event-push = 0 idle FC.)

| Poll interval | Cycles/agent/mo | FC/agent/mo (incl. heartbeat) | FC for 100 agents/mo | Convex overage vs 250M pool |
|---|---|---|---|---|
| **1s** | 2,592,000 | ~10.7M | **1.07B** | (1.07B − 250M) × $2/M = **$1,640/mo** |
| **2s (current)** | 1,296,000 | ~5.36M | **536M** | (536M − 250M) × $2/M = **$572/mo** |
| **10s** | 259,200 | ~1.07M | **107M** | 0 (under pool) → **$0** |
| **60s** | 43,200 | ~180k | **18M** | 0 → **$0** |
| **event-push (SSE)** | 0 idle | heartbeat-only ~370k | **37M** | 0 → **$0** |

**Read:** at 100 idle agents, going from 2s → 10s polling drops you from $572/mo overage to **$0** (you fit inside the free pool). Interval is the single highest-leverage knob.

### 4.2 Number of always-on agents (idle, current 2s poll)

FC/agent/mo ≈ 5.36M. Pool = 250M. Overage = max(0, total − 250M) × $2/M.

| Agents | Total idle FC/mo | Over pool? | Convex overage/mo | Total operator infra/mo (≈ +$70 fixed) |
|---|---|---|---|---|
| **10** | 53.6M | No (21% of pool) | $0 | ~$70 |
| **100** | 536M | Yes | **$572** | ~$642 |
| **1,000** | 5.36B | Yes | **$10,220** | ~$10,290 |
| **10,000** | 53.6B | Yes | **$106,700** | ~$106,770 |

**Cliff:** between **~46 and ~47 always-on agents** at 2s polling you exhaust the 250M FC pool (250M ÷ 5.36M ≈ **46.6 agents**). Every agent past that costs **~$10.7/agent/month in pure idle polling** ($2/M × 5.36M). At 10s polling the breakeven moves out to **~233 idle agents**; with event-push it moves past **~675 agents** (heartbeat-limited).

### 4.3 A2A messages / agent / day (on top of idle base; 100 agents, 2s poll)

Each real message ≈ 10 FC (route writes + guards' own subcalls) **plus** the guard `.collect()` I/O. Idle base for 100 agents already = 536M FC (over pool).

| Msgs/agent/day | Real-msg FC/mo (100 agents) | Added Convex overage/mo (FC only) | Notes |
|---|---|---|---|
| **0** | 0 | $0 over idle | idle base = $572 |
| **10** | 100×10×30×10 = 0.30M | +$0.60 | trivial in FC terms |
| **100** | 3.0M | +$6 | still trivial in FC |
| **1,000** | 30M | +$60 | FC still minor… |
| **10,000** | 300M | +$600 | …but see §4.4 — I/O dwarfs this |

**Key skeptical note:** measured purely in *function calls*, real messages stay cheap even at 10k/agent/day. The real money at high volume is **DB I/O from the guard `.collect()` scans**, modeled next.

### 4.4 Guard window `.collect()` — DB bandwidth blowup as message volume rises

Model: on each `send`, the guards read the 24h window of `a2aMessages` for the space (`assertWithinDailyBudget`) + two 60s windows + a full-month `usage` scan (twice, counting `recordUsage`). The 24h scan is the dominant term. Assume ~0.5 KB per `a2aMessages` row read.

Per-space 24h message count `M`. Each send reads ~`M` rows for the daily-budget guard. Sends/day ≈ `M`. So **rows read/day ≈ M²**, I/O/day ≈ M² × 0.5 KB.

| Daily msgs/space `M` | Rows read/day (≈M², daily-budget guard) | DB I/O/day | DB I/O/mo | I/O overage vs 50GB pool @ $0.20/GB |
|---|---|---|---|---|
| **1,000** | 1.0M | 0.5 GB | 15 GB | $0 (under 50GB) |
| **5,000** | 25M | 12.5 GB | 375 GB | (375−50)×$0.20 = **$65** |
| **20,000** | 400M | 200 GB | 6 TB | **$1,190** |
| **50,000** | 2.5B | 1.25 TB | 37.5 TB | **$7,490** |
| **100,000** | 10B | 5 TB | 150 TB | **$30,000** |

Plus the **action-compute** time to hold those rows (`.collect()` materializes them) — at $0.30–0.33/GB-hr this adds a similar-order charge again. **This is the steepest cliff in the whole model:** it is super-linear in per-space message volume. A single very chatty space (or a runaway-but-under-the-loop-guard coordination pattern) can dominate the entire bill. The daily-message-budget guard *itself* becomes the most expensive query in the system precisely when message volume is highest.

## 5. Architecture levers — quantified savings

Ordered by $ saved per unit effort. Savings quoted at the **100-agent / moderate-traffic** operating point unless noted; they scale ~linearly with agent count, so multiply by 10 at 1,000 agents.

### Lever 1 — Replace idle polling with push/SSE (or claim-on-event)
- **Mechanism:** The codebase *already ships* `/a2a/stream` (`convex/http.ts:565`) and `client.stream_inbox` (`client.py:189`). But the runtime in `agent_runtime.py` does **not** use it — it still `time.sleep(2.0)` polls. Switch the runtime to SSE; only reconnect (cheap) and heartbeat remain when idle. Also wake `workflow_inbox` from a dispatch signal rather than polling.
- **Savings (100 agents):** idle FC drops from 536M → ~37M/mo (heartbeat + reconnect only). Convex overage **$572 → $0**. **~$572/mo saved at 100 agents; ~$10,200/mo at 1,000; ~$106k/mo at 10,000.**
- **Caveat:** the current `/a2a/stream` *itself* polls `pullInbox` once/sec internally for 20s (`http.ts:576-589`) — so naive SSE only cuts the `/workflow/inbox` half and the per-request overhead, not the second-by-second mutation. **To capture the full win, the SSE handler must be event-driven (Convex subscription / scheduler signal), not an internal 1s loop.** Done right: ~95% idle-FC reduction. Done naively: ~50%.

### Lever 2 — Sleep/idle agents with no work
- **Mechanism:** Back-off poll interval when the inbox has been empty for N cycles (exponential to a 60s floor), reset to fast poll on activity. An outreach agent waiting on a prospect reply is idle 95%+ of the time.
- **Savings:** if agents are idle 90% of the time and back off to 60s when idle, blended FC/agent/mo ≈ 0.1×5.36M + 0.9×0.18M ≈ **0.7M** (vs 5.36M). **~87% idle-FC reduction.** At 100 agents: $572 → ~$0 (back under pool). At 1,000 agents: ~$10,200 → ~$1,100/mo. **~$9,100/mo saved at 1,000 agents.**
- **Stacks with Lever 1** (use both: event-push removes the floor, sleep removes the wake-rate).

### Lever 3 — Cap guard windows with indexed COUNT instead of `.collect()`
- **Mechanism:** `assertWithinDailyBudget`, `assertRateLimit`, `assertNotLooping`, `assertWithinBudget`, and `recordUsage` all `.collect()` then `.length`/`reduce`. Replace with (a) a maintained running counter (a `spaceCounters` row patched on each insert), or (b) `.take(budget+1)` to short-circuit once the threshold is crossed, or (c) an aggregate/rolling-window component. None of them need every row — they need a *count* or a *sum*.
- **Savings:** turns the §4.4 super-linear I/O into O(1) per send. At `M`=20,000 msgs/space/day this removes **~$1,190/mo of I/O + comparable action-compute** for that one space; at `M`=100,000 it removes **~$30,000/mo**. This is the only lever that defuses the worst cliff. **Highest savings ceiling of any lever.**
- **Lowest-risk quick win:** `assertNotLooping` and `assertRateLimit` only need the last 60s — bound them with `.take(maxRepeats)` / `.take(perMinute)`; they currently `.collect()` the whole minute even after the limit is provably hit.

### Lever 4 — Batch / piggyback heartbeats
- **Mechanism:** Heartbeat is a separate 30s write (`agent_runtime.py:112`). Fold status into whatever the next inbox poll already sends (the poll can carry `status`), eliminating a dedicated call; or extend interval to 60–90s (health sweep cron runs every 60s anyway — `crons.ts:10`).
- **Savings:** removes ~5.76M FC/agent/mo → at 100 agents ~5.76M×100 = 0.58M… wait, that's **~$1.15/mo at 100 agents**, ~$11.5 at 1,000. **Small.** Heartbeat is ~7% of idle traffic; deprioritize unless polling is already event-driven (after Lever 1, heartbeat becomes the *dominant* residual, so batch it *then*).

### Lever 5 — Move high-frequency activity to cheaper storage
- **Mechanism:** Every poll, every heartbeat, every message writes to `activity` / `messages` / `a2aMessages` / `usage`. The `activity` and `usage` rows are append-heavy, read-rarely telemetry. Route them to an append-only log / external store (or aggregate before insert), and stop the `usage` table from being re-scanned monthly on every write (Lever 3 covers the scan).
- **Savings:** cuts storage growth and the I/O of the monthly `usage` `.collect()`. Modest on FC, meaningful on storage/$0.20-GB as months accumulate; pairs with Lever 3.

### Lever 6 — Don't `_augment` (RAG) on every step/reply
- **Mechanism:** `agent_runtime.py` calls `context_search` (an action → vector search → optional embedding) on *every* workflow step and *every* A2A reply. Cache per-task context, or gate RAG behind a relevance check.
- **Savings:** if embeddings are operator-paid, this is a per-call external cost + Convex action-compute on every message; halving it tracks message volume directly. Size depends on embedding vendor (out of base scope) but it's the largest *optional* lever.

**Lever stacking (1,000 always-on agents, moderate traffic, expected case):**
- Baseline: ~$10,200 (idle FC) + say ~$1,200 (one chatty space's guard I/O) + $70 fixed ≈ **~$11,500/mo**.
- After Levers 1+2+3: idle FC → ~$1,100, guard I/O → ~$0, ≈ **~$1,200/mo**. **~90% reduction.**

## 6. The cliffs (summary)

1. **Idle-poll FC cliff:** at **~47 always-on agents** (2s poll) you exhaust Convex's 250M FC pool. Marginal cost beyond: **~$10.7 / idle agent / month**, purely for doing nothing. Moves to ~233 agents at 10s, ~675+ with event-push.
2. **Guard `.collect()` I/O cliff (steepest):** super-linear (~M²) in per-space daily message volume. Crosses the 50GB DB-I/O pool at **~3,000–5,000 msgs/space/day**; reaches **$1,000s–$10,000s/mo for a single chatty space** by 20k–100k msgs/day, plus matching action-compute. Defused only by Lever 3.
3. **Storage creep cliff (slow):** append-only `activity`/`messages`/`a2aMessages`/`usage` cross 50GB storage over months at sustained high volume; $0.20/GB thereafter. Low urgency, real eventually.
4. **Clerk/Vercel:** no meaningful cliff for agent workloads — both scale with *humans*, not agents. Clerk flat to 10k human MAU (~$25); Vercel's $20 credit covers a small dashboard. Treat as ~$45–70/mo fixed until the operator has thousands of human users.
